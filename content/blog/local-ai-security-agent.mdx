## Phase 1 — Picking Models (and getting it wrong first)

The original plan was dual MoE: a massive model for heavy lifting, a small fast model for easy tasks.

<Tabs color="#ff6600">
  <div label="What I tried first (wrong)">
    Started with **Qwen3.5-122B-A10B** in MXFP4_MOE quantization. The theory: 122B total params, only 10B active per token, so the active computation stays in VRAM and inactive MoE experts sit in RAM. Smart on paper.

```bash
# This was my first attempt — looked reasonable
nohup llama-server \
  -m Qwen3.5-122B-A10B-MXFP4_MOE.gguf \
  --n-gpu-layers 999 \
  --override-tensor "blk\..*\.ffn_.*_exps\.weight=CPU" \
  --ctx-size 32768 \
  --cache-type-k q4_0 --cache-type-v q4_0 \
  -fa --port 6969 > logs/qwen-heavy.log 2>&1 &
```

Result: **3.4 tokens per second**. GPU at 9% utilization. `nvidia-smi` showing 8.5GB used out of 23GB.
  </div>
  <div label="What was actually wrong">

The `--override-tensor` regex matched every single MoE expert weight tensor in every layer — 14.5GB VRAM idle, GPU at 9% utilization. MXFP4_MOE was retired by Unsloth and had poor CUDA kernel support for Qwen's Gated DeltaNet layers. Even if fixed, 122B at Q4 is ~61GB with 300 GB/s bandwidth ceiling = ~5 tok/s theoretical max. Physics don't work out.

  </div>
  <div label="What actually worked">

Switched to Qwen3.6-27B-Uncensored-HauhauCS-Aggressive-Q4_K_P — dense model, not MoE. 18GB fits entirely in VRAM, no CPU offloading needed. Gated DeltaNet means tiny KV cache (~0.7GB for 64k context). 14.5–18 tok/s, 5× faster than the 122B. HauhauCS-Aggressive removes safety filters that refuse security tasks. Has vision encoder (mmproj). Also runs Qwen3.5-9B via router mode for quick tasks.

  </div>

## Phase 2 — Setting Up llama.cpp

Building from source because I needed CUDA support and the latest router mode features.

```bash
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp

cmake -B build \
  -DGGML_CUDA=ON \
  -DGGML_CUDA_F16=ON \
  -DLLAMA_CURL=ON

cmake --build build --config Release -j$(nproc)
```

### The Router Mode Discovery

Originally tried to run two separate llama-server instances and proxy between them. That failed spectacularly. Then tried **llama-swap** — a third-party binary that hot-swaps models on demand. Also failed (502 errors, config path issues with Go template macros not expanding `~`).

Turns out llama.cpp shipped their own router mode in December 2025 and I just hadn't read the changelog. You start the server without `-m` and it auto-discovers models:

```bash
llama-server --models-dir $PATH_TO_PROJECT/models --models-max 1 --port 6969
```

<IdeaNode color="green">`--models-max 1` means only one model in VRAM at a time — LRU eviction kicks the old one out when a request comes in for a different model. Swap time is ~4 seconds for 9B, ~12 seconds for 27B. Way better than CPU inference.</IdeaNode>

## Phase 3 — The Smart Router (Python Proxy)

The llama.cpp router picks which model to load based on the `model` field in the request. So I wrote a thin FastAPI proxy that sits on port 7000, looks at the incoming prompt, and injects the right model name before forwarding to port 6969.

<CopyButton text={`# router.py — smart model selection based on prompt complexity
import httpx
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse, JSONResponse

app = FastAPI()
BACKEND = "http://localhost:6969"
LIGHT_MODEL = "Qwen3.5-9B-Uncensored-HauhauCS-Aggressive-Q6_K"
HEAVY_MODEL = "Qwen3.6-27B-Uncensored-HauhauCS-Aggressive-Q4_K_P.gguf"

HEAVY_PATTERNS = [
    "analyz", "exploit", "vulnerabilit", "bypass", "chain", "review",
    "audit", "explain", "reason", "plan", "strateg", "compare",
    "assess", "investigat", "what does", "is this vuln",
    "summarize", "report", "attack surface", "misconfig",
    "cve-", "poc ", "proof of concept", "write a", "generate a",
    "how would", "what could", "help me understand", "what is"
]

def pick_model(body: dict) -> str:
    requested = body.get("model", "")
    if requested in (LIGHT_MODEL, HEAVY_MODEL):
        return requested
    if requested == "light": return LIGHT_MODEL
    if requested == "heavy": return HEAVY_MODEL
    msgs = body.get("messages", [])
    if not msgs: return HEAVY_MODEL
    last = ""
    for m in reversed(msgs):
        if m.get("role") == "user":
            last = (m.get("content") or "").lower()
            break
    if len(last) > 400: return HEAVY_MODEL
    if any(p in last for p in HEAVY_PATTERNS): return HEAVY_MODEL
    return LIGHT_MODEL

@app.post("/v1/chat/completions")
async def completions(request: Request):
    body = await request.json()
    body["model"] = pick_model(body)
    if "max_tokens" not in body:
        body["max_tokens"] = 4096
    stream = body.get("stream", False)
    async with httpx.AsyncClient(timeout=700) as client:
        if stream:
            async def gen():
                async with client.stream(
                    "POST", f"{BACKEND}/v1/chat/completions",
                    json=body, headers={"Content-Type": "application/json"}
                ) as r:
                    async for chunk in r.aiter_bytes():
                        yield chunk
            return StreamingResponse(gen(), media_type="text/event-stream")
        r = await client.post(f"{BACKEND}/v1/chat/completions", json=body)
        return JSONResponse(r.json())`} label="COPY ROUTER.PY" />

### The 203/EXEC Nightmare

<Accordion title="The systemd bug that cost me 2 hours" color="#ff1144">
The router service kept failing with `status=203/EXEC`. Turned out the systemd service file was written using a heredoc where `$PYTHON_PATH` was never set, so the ExecStart ended up being literally `ExecStart=m uvicorn router:app` (just the letter "m"). Systemd was trying to run a binary called `m`.

**Fix:** Always use single-quoted heredoc `<< 'EOF'` when writing service files to prevent variable expansion, then hardcode paths.
</Accordion>

## Phase 4 — Goose Agent

<IdeaNode color="purple">Goose is the agentic layer. Think of it as the "body" that gives the LLM hands — it can run terminal commands, manage files, browse the web, query databases, spawn sub-agents. The LLM is just the brain; Goose is what lets it actually *do* things.</IdeaNode>

When you send it a task, it:
1. **Plans** — breaks the task into tool-call steps (heavy model)
2. **Acts** — calls MCP extensions (terminal, Qdrant, SearXNG, etc.)
3. **Observes** — reads tool output, decides next step
4. **Loops** — keeps going until task is done or it asks for input

### The SearXNG MCP Server

The MCP server at `/opt/mcp-searxng/server.py` doesn't exist by default — you have to write it. It's a simple JSON-RPC stdio server that wraps the SearXNG API:

<CopyButton text={`#!/usr/bin/env python3
import sys, json, requests, os

SEARXNG_URL = os.getenv("SEARXNG_URL", "http://localhost:4000")

def search(query: str, num_results: int = 10) -> str:
    try:
        r = requests.get(f"{SEARXNG_URL}/search",
            params={"q": query, "format": "json"}, timeout=15)
        results = r.json().get("results", [])[:num_results]
        out = []
        for i, res in enumerate(results, 1):
            out.append(f"[{i}] {res.get('title','')}\\n{res.get('url','')}\\n{res.get('content','')[:300]}")
        return "\\n\\n".join(out) if out else "No results."
    except Exception as e:
        return f"Search error: {e}"

TOOLS = [{"name": "searxng_search",
    "description": "Search the web via private local SearXNG",
    "inputSchema": {"type": "object",
        "properties": {"query": {"type": "string"}, "num_results": {"type": "integer", "default": 10}},
        "required": ["query"]}}]

def handle(msg):
    method = msg.get("method"); mid = msg.get("id")
    if method == "initialize":
        return {"jsonrpc":"2.0","id":mid,"result":{"protocolVersion":"2024-11-05","capabilities":{"tools":{}},"serverInfo":{"name":"searxng","version":"1.0"}}}
    if method == "tools/list":
        return {"jsonrpc":"2.0","id":mid,"result":{"tools":TOOLS}}
    if method == "tools/call":
        args = msg.get("params",{}).get("arguments",{})
        result = search(args.get("query",""), args.get("num_results",10))
        return {"jsonrpc":"2.0","id":mid,"result":{"content":[{"type":"text","text":result}]}}
    if method == "notifications/initialized": return None
    return {"jsonrpc":"2.0","id":mid,"error":{"code":-32601,"message":"Method not found"}}

for line in sys.stdin:
    line = line.strip()
    if not line: continue
    try:
        msg = json.loads(line)
        resp = handle(msg)
        if resp: print(json.dumps(resp), flush=True)
    except Exception as e:
        print(json.dumps({"jsonrpc":"2.0","id":None,"error":{"code":-32700,"message":str(e)}}), flush=True)`} label="COPY SEARXNG MCP" />

### Ingesting Security Knowledge

<Tabs color="#00aa44">
  <div label="CVE ingestion">
Daily delta — run via cron. Pulls NVD and CISA KEV:

```python
# Daily delta — run via cron
import requests, uuid, hashlib
from datetime import datetime, timedelta
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct
from sentence_transformers import SentenceTransformer

client = QdrantClient(url="http://localhost:6333")
embedder = SentenceTransformer("BAAI/bge-base-en-v1.5")

def upsert(text, meta):
    vec = embedder.encode(text, normalize_embeddings=True).tolist()
    uid = str(uuid.UUID(hashlib.md5(text.encode()).hexdigest()))
    client.upsert("security_kb", points=[
        PointStruct(id=uid, vector=vec, payload={"text": text, **meta})
    ])

def ingest_nvd_delta(days=1):
    end = datetime.utcnow()
    start = end - timedelta(days=days)
    url = f"https://services.nvd.nist.gov/rest/json/cves/2.0?lastModStartDate={start.strftime('%Y-%m-%dT%H:%M:%S.000')}&lastModEndDate={end.strftime('%Y-%m-%dT%H:%M:%S.000')}&resultsPerPage=2000"
    r = requests.get(url, timeout=60, headers={"User-Agent": "research/1.0"})
    for item in r.json().get("vulnerabilities", []):
        cve = item["cve"]
        desc = next((d["value"] for d in cve.get("descriptions",[]) if d["lang"]=="en"), "")
        if desc:
            upsert(f"{cve['id']}: {desc}", {"source":"nvd","cve_id":cve["id"],"type":"cve"})

def ingest_cisa_kev():
    r = requests.get("https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json", timeout=30)
    for v in r.json().get("vulnerabilities", []):
        text = f"ACTIVELY EXPLOITED: {v['cveID']}\nProduct: {v.get('vendorProject','')} {v.get('product','')}\nDescription: {v.get('shortDescription','')}"
        upsert(text, {"source":"cisa_kev","cve_id":v["cveID"],"type":"actively_exploited","priority":"critical"})
```

def ingest_mitre():

## Final Architecture

```
Telegram / Open WebUI / SSH terminal
         │
         ▼
  smart router :7000  ←── picks light/heavy based on prompt complexity
         │
         ▼
  llama.cpp router :6969  ←── loads/unloads model on demand (LRU)
         │
    ┌────┴────┐
    │         │
  27B GPU   9B GPU  (only one in VRAM at a time, ~4-12s swap)
  ~18GB     ~7GB
    │
    ▼
  Goose agent
  ├── developer (terminal, 600s timeout)
  ├── memory (persists to ~/.local/share/goose/memory/)
  ├── summon (spawn parallel subagents)
  ├── orchestrator (manage multiple sessions)
  ├── skills (auto-loads SKILL.md from ~/.config/agents/skills/)
  ├── qdrant MCP ──► Qdrant :6333 (CVEs, ATT&CK, H1 reports)
  ├── searxng MCP ─► SearXNG :4000 (live web search, private)
  └── fetch MCP (web content)
```

## Things That Went Wrong

<Accordion title="VRAM disaster — 8.5GB used out of 23GB" defaultOpen={false}>
Used `--override-tensor "blk\..*\.ffn_.*_exps\.weight=CPU"` which sent ALL MoE expert weights to CPU. Spent ages wondering why GPU utilization was 9%.

**Fix:** Don't use this flag for dense models. For MoE models if you must offload, use `--n-cpu-moe N` instead.
</Accordion>

<Accordion title="MXFP4_MOE quantization silently broken" defaultOpen={false}>
Unsloth retired MXFP4 from their XL quants. The format also had poor CUDA kernel support for Qwen's Gated DeltaNet layers, causing silent CPU fallback. 3.4 tok/s was the result.

**Fix:** Switch to UD-Q4_K_XL or Q4_K_P. Well-tested, proper CUDA support.
</Accordion>

<Accordion title="llama-swap 502 errors" defaultOpen={false}>
llama-swap config used Go template macros with `~` in paths. Go templates don't expand `~`. Every model process failed to start silently, returning 502.

**Fix:** Abandoned llama-swap entirely. Used llama.cpp's native router mode instead.
</Accordion>

<Accordion title="systemd 203/EXEC — the letter 'm'" defaultOpen={false}>
Wrote a systemd service file using a double-quoted heredoc with `$PYTHON_PATH` variable. Variable was never set, so ExecStart became `ExecStart=m uvicorn router:app`. systemd tried to run a binary called `m`.

**Fix:** Always use single-quoted heredoc `<< 'EOF'` to prevent variable expansion. Hardcode all paths.
</Accordion>

<Accordion title="qdrant/searxng silently not loading" defaultOpen={false}>
Both extensions in config.yaml were missing `enabled: true`. Goose skipped them without any warning or error.

**Fix:** Add `enabled: true` to every external MCP extension.
</Accordion>

<Accordion title="JSON truncation -32602 EOF error" defaultOpen={false}>
`code_execution` extension routes all tool calls through a code interpreter, producing huge intermediate outputs. Combined with the model's long thinking chains, responses exceeded token limits mid-JSON.

**Fix:** Disable `code_execution`. Add `/no_think` instruction to `.goosehints`.
</Accordion>

<Accordion title="nmap -p- timing out Goose sessions" defaultOpen={false}>
The agent ran `nmap -sV -p- localhost` which scans all 65,535 ports. With the developer extension's default 60-second timeout, the process got killed mid-scan.

**Fix:** Set `timeout: 600` on developer extension. Add `.goosehints` instruction to use `--top-ports 1000`.
</Accordion>

<Accordion title="Telegram bot token leaked" defaultOpen={false}>
Put the bot token directly in the systemd service ExecStart line. It's visible in `systemctl status` and `journalctl`.

**Fix:** Store tokens in separate file (`.env.telegram`, chmod 600). Reference via `EnvironmentFile`.
</Accordion>
